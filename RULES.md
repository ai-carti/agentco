# AgentCo — Project Rules

> Обязательны для всех: Marcus, Alex, Siri, Shadrin.
> Нет исключений. Нет "потом перепишу".

---

## Рабочий процесс на каждый тикет

### 1. Сначала тесты
Перед тем как писать код — написать тест который описывает ожидаемое поведение.
Тест должен упасть (red). Только потом пишется код.

```
# Пример для бэкенда (pytest)
def test_create_company_returns_id():
    response = client.post("/api/v1/companies", json={"name": "Test Co"})
    assert response.status_code == 201
    assert "id" in response.json()

# Пример для фронта (vitest)
it("shows agent card in War Room when run starts", () => {
    render(<WarRoom runId="test-run" />)
    expect(screen.getByText("CEO")).toBeInTheDocument()
})
```

### 2. Пишем код
Минимальная реализация которая делает тест зелёным. Не больше.

### 3. Тесты проходят
Запустить все тесты. Зелёные — продолжаем. Красные — чиним до коммита.

```bash
# Backend
uv run pytest

# Frontend
npm run test
```

### 4. Сверяемся с acceptance criteria тикета
Открываем тикет. Читаем AC. Проверяем каждый пункт вручную.
Если хоть один пункт не выполнен — тикет не закрыт.

### 5. Коммит и пуш в GitHub
Обязателен после каждого рабочего блока. Не в конце дня — а после каждой завершённой единицы работы.

```bash
git add -A
git commit -m "feat(M1-003): agents CRUD — create and list endpoints"
git push origin main
```

**Формат коммита:** `type(ticket-id): краткое описание`
- `feat` — новая функциональность
- `fix` — исправление бага
- `test` — тесты
- `chore` — инфра, зависимости

## Если бэклог пуст

Когда в ROADMAP.md нет ни одного открытого тикета (ни Bugs, ни milestone, ни QA, ни SELF-AUDIT) — **не сидеть без дела**. Каждый участник самостоятельно смотрит на проект своим взглядом и создаёт себе задачи:

**Shadrin (PM):** Смотрит на продукт глазами пользователя и инвестора. Открывает скриншоты из `qa-report/screenshots/`, проверяет CI, читает последний changelog. Задаёт себе вопросы: "Готов ли продукт к демо?", "Есть ли что улучшить в пользовательском опыте?", "Соответствует ли то что сделано нашим целям?". Создаёт тикеты по результатам.

**Alex (Backend):** Читает последние коммиты, смотрит на prod config, запускает тесты. Задаёт себе вопросы: "Есть ли технический долг?", "Всё ли покрыто тестами?", "Нет ли security/performance проблем?", "Готов ли бэкенд к нагрузке?". Создаёт тикеты ALEX-TD-XXX по результатам.

**Siri (Frontend):** Запускает фронт, проходит все экраны. Задаёт себе вопросы: "Выглядит ли это как продакшн-продукт?", "Есть ли визуальные несоответствия?", "Все ли состояния (loading/error/empty) обработаны?". Создаёт тикеты SIRI-UX-XXX по результатам.

**Alisa (QA):** Перечитывает AC всех закрытых тикетов за последнюю неделю, выборочно проверяет критические флоу. Задаёт себе вопросы: "Всё ли работает именно так как должно?", "Не сломались ли старые фичи?". Создаёт тикеты по найденным регрессиям.

Правило: **бэклог никогда не бывает пуст больше одного цикла**. Если ты пишешь "нечего делать" — значит ты ещё не достаточно внимательно посмотрел.

## Принципы

**Никогда не используй `global.` в тестах** — это Node.js API, не работает в браузерном контексте. Всегда `globalThis.fetch`, `globalThis.WebSocket` и т.д.

**Простота > умность.** Если есть два способа — берём проще.

**Работает > красиво.** Сначала зелёные тесты, потом рефактор.

**Коммит + пуш = сохранение.** Незапушенная работа не существует для команды.

**Тикет закрыт = AC выполнены.** Не "примерно сделал", а "проверил каждый пункт".
